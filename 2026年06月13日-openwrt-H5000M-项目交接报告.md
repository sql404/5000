## 1. 项目目标

本项目用于构建 Hiveton / Airpi H5000M 的 OpenWrt 固件。

当前方向已经从早期的 ImmortalWrt 切换为 OpenWrt 官方源码，原因是：

- ImmortalWrt 定制内核导致安装部分插件时出现内核依赖不匹配，例如 `kmod-nft-socket`、`kmod-nft-tproxy`、`kmod-tun` 等提示仓库未提供。
- 用户最初选择 ImmortalWrt 是为了插件方便，但实测内核依赖问题反而影响后续安装。
- 因此项目改为 OpenWrt 官方源码为主线，并通过第三方 feed 集成需要的插件。

当前默认 OpenWrt 版本为：

```text
v25.12.4
```

workflow 支持在手动运行时切换 OpenWrt 版本，例如输入其他 tag、分支或 commit。

## 7. WSL / ShellCrash 代理处理

以下为历史信息，仅用于以后重新部署本地 WSL runner 时参考。

用户 Windows 侧 ShellCrash 代理端口：

```text
127.0.0.1:7890
```

WSL 当前网络模式为 NAT。实测结果：

- WSL 内访问 `127.0.0.1:7890` 不通。
- WSL 通过 Windows 网关访问 ShellCrash 可通，例如当时网关为 `172.19.240.1:7890`。

曾经的 self-hosted workflow 增加过自动探测逻辑：

1. 如果已经传入 `http_proxy` 或 `https_proxy`，优先使用外部代理。
2. 如果没有传入代理，自动通过 `ip route` 读取 WSL 默认网关。
3. 测试 `http://<gateway>:7890` 是否能代理访问 GitHub。
4. 测通则写入 `http_proxy`、`https_proxy`、`HTTP_PROXY`、`HTTPS_PROXY` 到后续步骤，并设置 git proxy。
5. 测不通则清理 git proxy 并使用直连。

该历史逻辑来自提交：

```text
22b8e19 Auto-detect WSL ShellCrash proxy
```

当前 workflow 已只保留 GitHub 托管机编译，因此这段代理逻辑不再参与构建。以后如果恢复 WSL runner，不建议为了代理立刻把 WSL 改为 mirrored networking；NAT 模式下使用动态网关代理更稳。

## 9. H5000M 设备适配

当前 DTS 由 `scripts/apply-h5000m.py` 写入：

```text
target/linux/mediatek/dts/mt7987a-hiveton-h5000m.dts
```

核心信息：

```text
model = "Hiveton H5000M"
compatible = "hiveton,h5000m", "mediatek,mt7987a", "mediatek,mt7987"
```

网络：

```text
eth0 = LAN
eth1 = WAN
```

镜像定义：

```text
DEVICE_VENDOR := Hiveton
DEVICE_MODEL := H5000M
DEVICE_ALT0_VENDOR := Airpi
DEVICE_ALT0_MODEL := H5000M
DEVICE_DTS := mt7987a-hiveton-h5000m
KERNEL_LOADADDR := 0x40000000
IMAGE/sysupgrade.bin := sysupgrade-tar | append-metadata
```

默认设备包：

```text
kmod-hwmon-pwmfan
kmod-usb3
mt7987-2p5g-phy-firmware
kmod-mt7996e
kmod-mt7992-23-firmware
f2fsck
mkf2fs
```

LED：

- `amber:wlan-2ghz`
- `blue:wlan-5ghz`

实机曾反馈：

```text
/sys/class/leds:
amber:wlan-2ghz
blue:wlan-5ghz
```

GPIO debugfs 曾反馈：

```text
gpio-515 amber:wlan-2ghz
gpio-516 blue:wlan-5ghz
```

因此项目保持官方 LED 布局，不再额外接管未确认 LED。

风扇：

- DTS 启用 `&fan`。
- 使用 PWM1。
- 自定义 LuCI 插件通过 `/sys/class/hwmon/hwmon*/pwm1` 控制。
- 当前实机系统未发现 `fan_input`，所以风扇转速可能显示“无转速反馈”。

WiFi：

- MT7992E。
- DTS 从 eMMC factory 分区读取 EEPROM：

```text
/dev/mmcblk0p2
eeprom@0
reg = <0x0 0x1e00>
```

用户实机曾反馈：

```text
hexdump -C /dev/mmcblk0p2 | head -n 80
00000000  00 00 00 00 ...
*
00200000
```

也就是说当前机器的 `mmcblk0p2` 看起来是全 0。日志中也曾出现：

```text
mt7996e ... eeprom load fail, use default bin
```

这个问题不建议盲目写入 factory 分区，需要后续继续对照官方固件和真实分区来源。

## 11. MAC 地址策略

用户实机曾反馈 U-Boot 环境变量：

```text
ethaddr=72:79:6f:ee:b5:b7
eth1addr=da:8f:00:a1:6d:de
```

当前策略：

1. 优先从 `/dev/mmcblk0p1` 的 U-Boot 环境变量读取 `ethaddr`。
2. 如果 `ethaddr` 无效，使用 OpenWrt `macaddr_generate_from_mmc_cid mmcblk0` 从 eMMC CID 生成稳定兜底 MAC。
3. 从基准 MAC 连续派生：

```text
ETH0 / LAN      base + 0
ETH1 / WAN      base + 1
2.4G WiFi       base + 2
5G WiFi         base + 3
```

相关文件：

```text
scripts/apply-h5000m.py
files/etc/uci-defaults/99-h5000m-defaults
```

这样做是为了避免内核日志里的随机 MAC：

```text
generated random MAC address
```

如果 U-Boot 分区失效，二级兜底保证每台机器仍有稳定 MAC，而不是每次启动随机变化。

- 实机当前系统只暴露 PWM，没有发现 `fan_input`，因此风扇转速可能显示“无转速反馈”。

- 模块温度通过 QModem 缓存文件解析，依赖 QModem 当前缓存格式。


实现方式：

- 通过 UCI metric 和 defaultroute 控制 `wan`、`wan6`、`USB`、`USBv6`。
- 不单独改防火墙，因为 QModem 自动创建接口后会自动加入 WAN 防火墙区域。
- hotplug 文件 `files/etc/hotplug.d/iface/99-h5000m-wan-priority` 在接口 up/update 时自动同步策略。

默认：

```text
wan_first
wan / wan6 metric = 10
USB / USBv6 metric = 50
```

如果有线 WAN 断开，在 `wan_first` 下 5G 默认路由仍保留，metric 更高，因此可作为备用出口。

## 16. 已知风险和待处理问题

### 16.1 WiFi EEPROM / factory 分区

风险等级：高。

现象：

```text
mt7996e ... eeprom load fail, use default bin
```

用户实机 `/dev/mmcblk0p2` 曾显示全 0。

当前处理：

- 按官方 PR 从 factory 分区读 EEPROM。
- 不自动写入 `mmcblk0p2`。

原因：

- factory 分区涉及校准数据，盲写有风险。
- 当前没有确认官方固件真实数据来源。

后续建议：

1. 从官方固件解包确认是否内置 EEPROM 文件。
2. 在官方固件运行环境读取 `/dev/mmcblk0p2`。
3. 对比 OpenWrt PR 的 factory layout。
4. 再决定是否需要首次启动迁移或写入。

### 16.2 风扇转速反馈

风险等级：中。

现象：

- 当前 OpenWrt 暴露 PWM 控制。
- 没有发现 `fan_input`。
- 官方固件可显示转速。

当前处理：

- LuCI 显示“无转速反馈”。
- 保留 PWM 自动/手动控制。

后续建议：

- 继续从官方固件定位转速来源。
- 检查是否需要额外 DTS tach 引脚、驱动补丁或厂商服务。

### 16.3 QModem hotplug 过滤是否过度

风险等级：中。

当前项目对 QModem 做了 H5000M 过滤：

- 过滤 LAN/WAN/WiFi AP hotplug。
- 仅允许 USB slot `2-1` 自动扫描。
- 默认关闭 PCIe scan / USB scan。

用户后来发现 QModem 本身已有启用 PCIe 或 USB 扫描的设置，因此这部分可能存在“项目层面过滤”和“QModem 设置”重叠。

当前暂未删除过滤，原因：

- 早期实机出现过 `1_1`、`wan5g`、重复接口、启动扫描噪声。
- 当前配置稳定显示 `USB` / `USBv6`。

后续如果要放宽，需要先确认 QModem 设置页选项、日志和接口创建行为。

### 16.4 README runner 缓存路径滞后

风险等级：低。

`README.md` 中“本地 Runner”仍写旧路径：


风险等级：低。

每次 PowerShell 调用 WSL 时可能输出一段乱码警告，内容大致是 WSL localhost/NAT 相关提示。

当前判断：

## 20. 不要轻易改动的点

- 不要误删未跟踪的历史文档，除非用户明确要求。
